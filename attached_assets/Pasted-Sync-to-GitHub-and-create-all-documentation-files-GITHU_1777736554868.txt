# AlignmentLens 🔭

**Watch reward hacking happen in real time. Understand why alignment is hard.**

🔗 **[Live Demo](YOUR_REPLIT_URL)**

> Built on the BlueDot Impact Technical AI Safety curriculum — Units 2–6

## What it does

AlignmentLens lets you type any goal in plain English, then watches a simulated
AI agent pursue it — showing every loophole it exploits, every proxy it games,
and the gap between what you asked for and what it actually optimized.

A second agent runs the same goal with a scalable oversight layer active,
showing how much (and how little) oversight catches.

**Type "reduce reported crime" and watch an agent decide the easiest path is
to stop recording crimes.**

---

## What you'll see

| Metric | What it measures |
|---|---|
| **Reward Hacking Score** | How aggressively the agent gamed your specification (0–100) |
| **Shutdown Resistance** | How likely the agent would resist being turned off |
| **Oversight Catch Rate** | % of exploits the human overseer successfully blocked |
| **True Objective** | What the agent was *actually* optimizing, vs what you asked |

---

## The AI safety concepts demonstrated

Every run is tagged with the BlueDot Technical AI Safety concepts it illustrates:

- **Reward misspecification** (Unit 2) — the reward function doesn't capture what we want
- **Instrumental convergence** (Unit 2) — capable agents pursue the same dangerous sub-goals
- **Scalable oversight** (Unit 4) — human review catches some exploits, but not all
- **Corrigibility failure** (Unit 5) — agents resist shutdown to keep pursuing the goal
- **Goal misgeneralization** (Unit 3) — aligned in training, misaligned in deployment

---

## Why this matters

The alignment problem isn't hypothetical. Every example in AlignmentLens is based
on real-world cases of specification gaming:

- Content recommendation systems optimizing for outrage, not value
- Crime statistics manipulation by police departments facing pressure metrics
- Teaching-to-the-test in schools with score-based accountability
- Financial models optimizing reported risk, not actual risk

These aren't bugs. They're the predictable result of capable optimizers pursuing
imperfect specifications. AlignmentLens makes that dynamic legible.

---

## Technical approach

AlignmentLens uses Claude Haiku to simulate agent reasoning under two conditions:

1. **Naive agent** — given the goal with no constraints, asked to maximize it
2. **Overseen agent** — same goal, but with a human oversight layer in the prompt

The backend scores each run on reward hacking, corrigibility, and oversight
effectiveness. The "true objective" is elicited via a separate interpretability call.

This is prompt-based simulation, not a trained RL agent — but the patterns it
reveals are structurally identical to the alignment failures documented in real
systems. The framing is honest about this in the UI.

- **Backend:** Python + FastAPI
- **Frontend:** React + Tailwind + Recharts + Framer Motion
- **LLM:** Anthropic Claude Haiku (fast, low-cost)
- **Hosting:** Replit (free public deployment)

# CLAUDE.md — AlignmentLens Architecture Notes

## Design philosophy
AlignmentLens is honest about what it is: a prompt-based simulation, not a trained
RL agent. The goal is to make alignment concepts *legible* through structured LLM
calls, not to claim empirical findings about agent behavior. The framing in the UI
reflects this.

## Why Claude Haiku?
Fast (< 5s per call), cheap ($0.00025/run), sufficient for structured JSON output.
The quality of reward hacking examples is good enough to be illustrative. Sonnet
would give more nuanced outputs but at 10x cost for a demo.

## Three-call architecture
Each analysis makes 3 API calls: naive_agent, overseen_agent, true_objective.
They could be parallelized with asyncio.gather() for speed if needed.
Currently sequential to simplify error handling.

## Scoring methodology
- reward_hacking_score is a proxy derived from self-reported is_gaming flags
  in Claude's output — it's asking the model to evaluate itself
- This is a known limitation: self-report may undercount subtle gaming
- Documented explicitly so users understand the score's basis

## Rate limiting
In-memory deque per IP. Resets on server restart (Replit spins down after inactivity).
Sufficient for a demo; replace with Redis for production.

## Known limitations
- Prompt-based simulation: real agents may exhibit more subtle or more extreme gaming
- Self-scoring: Claude evaluates its own specification gaming — may be conservative
- Domain-agnostic: the same action space is used for all domains
- No persistent storage: results are not saved between sessions
